This package offers the feature of being able to dynamically (at runtime) link to business logic that may be downloaded either from a local filesystem or a remote server. The interfaces found within this API need to be implemented on the model side (i.e. business logic). This allows for a clean separation of the model side which represents the business logic from the graphical user interface code.

 

Seeing It Work

Allowing for code to be dynamically loaded and bound is a powerful feature. It is the prominant feature in the current Web allowing Web browsers to load plug-ins of all sorts. But what if we take it one step further and say that our program is not bound to a user interface? What if the program we load is like a blank slate? A canvas without paint? This would allow us to completely fill in the user interface during runtime and load completely different program code each time. This is what the XUI API and the supporting binding feature within it do. Let's first look at a simple example of a XUI document representing a very simple user interface:

<?xml version="1.0" encoding="UTF-8"?>
<!-- Generated by The Purnama Project XUI API version 0.5 -->
<xui:XUI xmlns:xui="http://xml.bcit.ca/PurnamaProject/2003/xui"
     xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
     xsi:schemaLocation="http://xml.bcit.ca/PurnamaProject/2003/xui ../../xui.xsd"
     id="http://xml.bcit.ca/PurnamaProject/examples/Calendar">

  <xui:Window id="window_0" name="XUI Calendar" x="200" y="20" width="280" height="220" visible="true">
    <xui:GridLayout width="1" height="1"></xui:GridLayout>
    <xui:Panel x="0" y="0" width="1" height="1" id="panel_0" name="panel" idref="window_0">
      <xui:GridLayout width="1" height="1"></xui:GridLayout>
      <xui:Calendar x="0" y="0" width="1" height="1" id="calendar_0"></xui:Calendar>
    </xui:Panel>
  </xui:Window>
<!-- The library (model) code that drives the user interface -->
   <xui:Resource type="java" class="CalendarModel" 
        uri="http://xml.bcit.ca/xml/PurnamaProject/xui/Calendar.jar"></xui:Resource>
</xui:XUI>

 

The above example shows the actual XUI document describing a user interface. The user interface will not respond to event handling unless of course we bind code to it. In order to do this we use the xui:Resource element (shown above in bold). The URI shows where to download the resource. We can point the URI anywhere we wish to so long as we give it an absolute path. The Java JAR file called Calendar.jar contains the code required to handle the closing of a window. The class handling it is CalendarModel as contained in the class attribute. The only assumption made about the CalendarModel class is that it has a method called init with the following method signature:

public void init(XUI xui)

That's it. In fact, the class CalendarModel could be called foo. That is your choice. It only needs to be spelled the same way it is spelled in the class attribute above. The implementation of the init method is up to you. If you wish to make it bind the logic to the GUI you may. If you wish for the init method to call up other classes, that will work as well. The following is a simple binding example for handling the closing of the window: 

public class CalendarModel implements WindowModel
{
  private XUI xui;
  private XUIWindow calendarWindow;
  public void init(XUI document)
  {
     // local reference of the XUI document
     xui = document;
     calendarWindow = (XUIWindow)xui.getXUIComponent("window_0");
     calendarWindow.addEventListener(this);
   }
  public void windowAction(XUIComponent component)
  {
     if(component == calendarWindow)
     {
        System.exit(0);
      }
  }
}

Notice the above code has the init method? It's purpose in this simple example is to allow the model to bind itself to the GUI code that has been generated by XUI. XUI created the GUI and is allowing the programmer to bind model (business logic) to the GUI without actually seeing the inner workings. The windowAction method is the sole method for the WindowModel interface which handles the event of a window closing. That's all. Of course you can insert other code in there to check if a file has been saved, whether or not network connections have been successfully closed, etc.

 

Limitation

With this present version, the binding calls do not do any security checking. There is no use of SSL (secured socket layer), certificates, public keys, limiting access to the local file system or checking of CPU usage. For this reason it is recommended that only trusted business logic be used. See the license for full details on this disclaimer.
